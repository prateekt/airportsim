Integration Problems Login/Registration/User Editing

 

This part of integration involved Andrew Tio, Eric Quillen and me. We were combining Andrew and my already integrated back-end with Eric's GUI.

 

Many of the early problems were naming conventions for methods and deciding what information we actually had to pass to each other. One of the bigger issues we faced was matching Eric's HW3Personnel user class to our UserSchema. Though the classes held the same data we did not want to change all the code in either file. We ended up translating the HW3Personnel class into a UserSchema in the LoginAgent. This was not very difficult and allowed Andrew's LoginAgent to pass a UserSchema to the database.

 

Another issue we faced was combining a scheduler driven LoginAgent with a non-scheduler GUI. Since none of LoginAgent's classes returned values we had to call a method in the GUI class to update the list of users upon changes or complete the login process. These methods are called printList in HW3RegistrationPanel and msgLoginResult in HW3LoginPanel, respectively. In this manner we were able to let the LoginAgent respond to the GUI on its own time rather than the GUI's schedule.

 

This scheduler response design presented an issue which we had to figure out. The user lists were not updating until the next action after a change. We discovered that the issue was the placement of the refresh action at the top spot in the LoginAgent's scheduler. Therefore if the GUI requested an add and a refresh immediately following, the LoginAgent would service the refresh first then the add. This problem was easily fixed by moving the refresh action to lowest priority in the scheduler.
