105 Design Plan

 

The first priority in creating Homework 4 is fragmenting the problem into smaller pieces.  We have decided to break the assignment into four parts:  Rendering runways, rendering airplanes, creating lists, and creating the Supervisor and Ground Control panels.

Rendering Runways – this should be done so that the user can view the layout of the airport.  Labels, a scale, and differentiating between taxiways and runways should be able to be done by eye.  We will need to create a method that can paint a runway.  The dimensions will be 680 x 512 pixels.

Rendering Airplanes – similarly to runways, the movement of these objects must be able to be tracked by eye.  The panel will be the same one used for rendering runways.  Different color schemes can be used for different statuses.  Colors will change in accordance to status changes: green, yellow, orange, red for good, fair, poor, and immanent respectively.  Color can also be used to indicate an arriving, departing, or taxiing airplane.

Creating Lists – Three lists will be necessary for displaying risks, mitigations, and actions.  These may change color according to status.  The highest priority risk should be placed at the top of the list, while the highest suggested mitigation and action will be displayed at the top of those lists.  When clicking on a risk, the mitigation list will populate for that risk.  When a mitigation is clicked, the actions associated with it will be displayed in the action list.

Creating Panels – Since the supervisor will need the ability to change views to any type of user, a tabbed pane would work well to switch panes.  The Ground Control user should be able to see a runway with the movement of airplanes, along the lists of risks, mitigations, and actions.  These should be displayed in a way that is natural for the user to react to situations that may occur.

 

            Once we have fragmented the large problem into smaller, more maintainable problems, we can work on other functionalities.  The login process needs to be simple, and robust from security breaches.  We will need to time step through the simulation with a button.  We must make this program as scalable as possible since we will have more features in the next design.

 

            Integration – we must know how data is to be passed from the backend to the GUI.  Once we have obtained the data to be displayed, the GUI side of the program is to keep its own copy as the most recent data and display it.

            Our priority in version 2.0 was to have as much as possible to be dynamic.  This would enable the system to be scalable for growth and have different scenarios function without any hard-coding.  All airplane paths would be decided by actions chosen by an action list from a menu.  Of course, everything was to be fully integrated with the 201 backend.  Airplanes are to dock at a hangar after landing on a runway, and leave a hangar and takeoff from a runway.

            Other issues we will work on in version 2.0 are cleaning up our code and making the GUI look attractive.  We will add a wind and weather box, selection highlighting for airplanes, a local control panel, a way to pass responsibility from ground control to local control and vice versa, and a north pointer.  Dynamic labeling will be implemented for airplanes, buildings, taxiways, and menu items.
